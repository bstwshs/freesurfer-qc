# Отчёт о реализации прототипа QC FreeSurfer

## 1. Введение

**Цель проекта:** разработать прототип программного модуля для автоматического контроля качества (Quality Control) результатов обработки структурных МРТ-снимков программным комплексом FreeSurfer.

**Задачи прототипа:**
- Реализовать парсинг выходных файлов FreeSurfer (aseg.stats, aparc.stats).
- Реализовать сравнение измеренных значений с нормативными через расчёт Z-score.
- Реализовать трёхуровневую классификацию: OK, Check, Bad.
- Реализовать визуализацию среза мозга при обнаружении брака.
- Предоставить веб-интерфейс для удобного взаимодействия с модулем.

**Актуальность:** ручная проверка результатов FreeSurfer экспертом трудоёмка. Автоматический скрининг позволяет сократить нагрузку на специалистов, оставляя им только пограничные и проблемные случаи.

## 2. Реализованная функциональность

### 2.1. Модуль ввода данных (input_handler.py)

Функция `check_input_folder(path)` проверяет наличие трёх обязательных файлов в папке субъекта:
- `aseg.stats` — статистика объёмов подкорковых структур;
- `aparc.stats` — статистика толщины кортикальных регионов;
- `brain.mgz` — NIfTI-изображение мозга.

Возвращает `(True, "")` при успехе или `(False, сообщение)` с указанием отсутствующих файлов.

### 2.2. Модуль парсинга (parser_engine.py)

Функция `parse_stats(folder_path)` читает оба .stats файла, пропускает строки-комментарии (начинающиеся с `#`), извлекает:
- из aseg.stats: `StructName` → Region, `Volume_mm3` → Value, Type = "Volume";
- из aparc.stats: `StructName` → Region, `ThickAvg` → Value, Type = "Thickness".

Возвращает объединённый pandas DataFrame с колонками `Region, Value, Type`.

### 2.3. Ядро валидации (qc_core.py)

Функция `run_qc(df, norms_path)`:
- Загружает `norms.csv` (колонки Region, Type, Mean, Std).
- Выполняет слияние данных с нормами по `Region` и `Type`.
- Вычисляет Z-score = (Value − Mean) / Std.
- Классифицирует: |Z| < 2 → OK, 2 ≤ |Z| < 3 → Check, |Z| ≥ 3 → Bad.

Возвращает DataFrame с добавленными колонками: `Mean, Std, Zscore, Verdict`.

### 2.4. Модуль визуализации (visualizer.py)

Функция `generate_slice(brain_mgz_path, region_name)`:
- Загружает NIfTI-файл через NiBabel.
- Извлекает средний осевой срез (по оси z).
- Отображает срез в оттенках серого (colormap="gray").
- Добавляет красную прямоугольную рамку в центре среза как имитацию подсветки проблемной зоны.
- Сохраняет изображение в `slice_output.png`.

### 2.5. Веб-интерфейс (app.py)

Streamlit-приложение с элементами:
- Текстовое поле для ввода пути к папке субъекта.
- Кнопка «Запустить QC».
- Вывод: таблица результатов, гистограмма Z-score, столбчатая диаграмма вердиктов, сводная таблица.
- При наличии Bad — автоматическая генерация и отображение среза мозга.

### 2.6. Взаимодействие модулей (поток данных)

```
Пользователь → [app.py] → check_input_folder()
                        → parse_stats()
                        → run_qc()
                        → generate_slice()  (только при Bad)
                        → отображение результатов
```

## 5. Анализ

### 5.1. Что работает

1. **Генерация синтетических данных** — скрипт создаёт файлы, структурно близкие к реальным выходным данным FreeSurfer (имена колонок, формат строк, структура заголовков).
2. **Парсинг .stats файлов** — корректно пропускаются строки-комментарии, извлекаются имена структур и числовые значения.
3. **Расчёт Z-score** — сравнение с нормами работает как для объёмов, так и для толщины; корректно обрабатывается ситуация отсутствия норм для региона.
4. **Трёхуровневая классификация** — вердикты OK/Check/Bad присваиваются по порогам |Z| = 2 и |Z| = 3.
5. **Веб-интерфейс Streamlit** — все элементы управления работают, результаты отображаются в структурированном виде.
6. **Визуализация среза** — NiBabel загружает .mgz, Matplotlib генерирует изображение среза с подсветкой.

### 5.2. Что требует доработки

1. **Поддержка реальных файлов FreeSurfer.** Синтетические данные упрощены. Реальные .stats файлы могут содержать другие разделители, разное число колонок, дополнительные секции. Потребуется доработка `parser_engine.py` после получения образцов реальных данных.

2. **Топологические метрики.** В текущем прототипе не анализируются топологические дефекты (число дыр, пересечений поверхности). В методологическом разделе это заявлено как часть гипотезы — требуется отдельный модуль анализа лог-файлов FreeSurfer.

3. **Батчевая обработка.** Прототип обрабатывает одного субъекта за раз. Для массового скрининга нужна пакетная обработка (очередь папок, сводный отчёт по группе).

4. **Улучшенная визуализация.** Текущая красная рамка — имитация. В реальном проекте необходимо сопоставлять регион с координатами в атласе (например, FSaverage) и подсвечивать конкретную анатомическую область на поверхности или в объёме.

5. **Калибровка норм.** Нормативные значения в `norms.csv` — примерные. Для практического применения требуется загрузка норм из референсных исследований или калибровка на локальной выборке здоровых испытуемых с учётом возраста, пола и параметров томографа.

6. **Обработка граничных случаев.** Не полностью покрыты сценарии: частичное повреждение .stats файла, нестандартные имена регионов, отсутствие brain.mgz при наличии Bad-вердикта.

## 6. Заключение

Разработанный прототип успешно реализует базовый пайплайн контроля качества результатов FreeSurfer:
- парсинг выходных файлов,
- сравнение с нормами через Z-score,
- трёхуровневую классификацию,
- визуализацию проблемных зон,
- веб-интерфейс для конечного пользователя.

Прототип соответствует поставленным задачам в рамках задания 5П. Модульная архитектура позволяет независимо дорабатывать каждый компонент (парсер, ядро QC, визуализатор) при переходе на реальные данные. Код готов к размещению на GitHub и демонстрации.
